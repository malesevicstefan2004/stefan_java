1. Aufgabenbeschreibung

Im Rahmen dieser Aufgabe war ein klassisches Memory-Kartenspiel als interaktive JavaFX-Desktopanwendung zu implementieren. Das Spiel soll vollständig nach dem MVC-Entwurfsmuster (Model–View–Controller) strukturiert sein und das Observer-Pattern für die Kommunikation zwischen Modell und View nutzen.

3. Architektur

3.1 MVC-Entwurfsmuster

Die Anwendung ist strikt nach dem MVC-Muster aufgebaut. Jede Schicht hat eine klar definierte Verantwortung und kennt die darunterliegenden Schichten nicht:

SchichtKlassen / DateienVerantwortung
Model GameModel, Card, Player,
GameConfig, Theme, CardState		 Spiellogik, Zustand der Karten und Spieler; kennt die View nicht
View MenuView.fxml, GameView.fxml, style.css Deklarative Darstellung der Benutzeroberfläche, enthält keine Logik
Controller MenuController, GameController Verbindet View und Model; reagiert auf Benutzeraktionen und Update-Events

3.2 Observer-Pattern

Das Model kommuniziert mit dem Controller ausschließlich über das GameModelListener-Interface. Der GameController registriert sich beim Start mit model.addListener(this) als Observer und wird bei jedem relevanten Spielereignis benachrichtigt:

Methode (Interface)Auslöser
onCardFlipped()Eine Karte wurde umgedreht
onPairMatched(r1,c1,r2,c2)Zwei Karten bilden ein Paar
onPairMismatched(r1,c1,r2,c2)Zwei Karten passen nicht zusammen
onCardsHidden()Nicht passende Karten wurden wieder umgedreht
onGameOver()Alle Paare wurden gefunden

5. Spiellogik

5.1 Spielablauf

Jede Runde besteht aus dem Aufdecken von genau zwei Karten. Das Modell verwaltet den Zustand intern über die Felder flippedCount, firstRow/Col und secondRow/Col: 

Klick auf Karte
  └─► flipCard(row, col)
        ├─ flippedCount == 0 → erste Karte merken, onCardFlipped()
        └─ flippedCount == 1 → zweite Karte, totalMoves++, evaluatePair()
               ├─ pairId gleich → MATCHED, score++, onPairMatched()
               │       └─ alle Paare? → onGameOver()
               └─ pairId verschieden → onPairMismatched()
                       └─ (nach 1500 ms) hideCards() → Spielerwechsel, onCardsHidden()

5.2 Spielerwechsel

Der Spielerwechsel erfolgt nur bei einem Mismatch (in hideCards() durch currentPlayerIndex = (currentPlayerIndex + 1) % players.size()). Bei einem Match bleibt der aktive Spieler am Zug – ein Merkmal vieler klassischer Memory-Varianten.

5.3 Spiel-Ende

Sobald matchedPairs == (rows × cols) / 2, wird onGameOver() ausgelöst. Der GameController zeigt einen Alert-Dialog mit:

  • 1-Spieler: Glückwunsch-Meldung mit Anzahl der Züge
  • 2-Spieler: Gewinner oder Unentschieden mit Punktestand (z.B. 6 : 2)

Lombok-Einsatz

Lombok wird in allen Modellklassen eingesetzt, um Getter und Setter automatisch zu generieren. Dies reduziert den Code erheblich und verbessert die Lesbarkeit: 

@Getter          // generiert get-Methoden für alle Felder
@Setter          // generiert set-Methoden für alle Felder
public class GameConfig { ... }

@Getter          // nur Getter (unveränderliche Objekte)
public class GameModel { ... }

10. Fazit

Das Memory-Spiel wurde vollständig nach den gestellten Anforderungen implementiert. Die MVC-Architektur sorgt für eine klare Trennung der Verantwortlichkeiten: das Modell enthält ausschließlich Spiellogik ohne jegliche Abhängigkeit zur View, die FXML-Views sind rein deklarativ, und die Controller verbinden beide Schichten über das Observer-Interface GameModelListener.

Durch den Einsatz von Lombok wird Boilerplate-Code in den Modellklassen minimiert. Maven übernimmt Build, Dependency-Management und die Erzeugung der JavaDoc-API-Dokumentation. Das Spiel unterstützt drei Boardgrößen, drei Symbolthemen sowie Einzel- und Zweispieler-Modus mit vollständiger Punkteauswertung und Endbildschirm.
